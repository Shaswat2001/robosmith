    Stage 7: Delivery¶

    The delivery stage packages all artifacts from the pipeline run into a clean output directory. It produces a human-readable report, saves the reward function as a standalone Python file, records a video of the trained policy, and optionally pushes the checkpoint to HuggingFace Hub.

    Why this stage exists¶

    A trained policy checkpoint alone isn't very useful. You need to know what reward function was used, how well the policy performed, what decisions the pipeline made, and ideally see a video of the robot in action. The delivery stage ensures every run produces a complete, self-contained package that you can share, reproduce, or build on.

    Delivery always runs — even if the pipeline failed partway through. A failed run still produces a report explaining what happened and which artifacts are available.

    What gets produced¶

    Every run creates a directory in robosmith_runs/:

    robosmith_runs/run_20260322_231140_57f0f9/
├── reward_function.py     # The evolved reward function
├── policy_ppo.zip         # Trained model checkpoint
├── eval_report.json       # Evaluation metrics
├── policy_rollout.mp4     # Video of the trained policy
├── report.md              # Human-readable run summary
├── run_state.json         # Full pipeline state (for debugging)
└── task_spec.json         # Parsed task specification

    reward_function.py¶

    The evolved reward function saved as a standalone Python file. Includes a docstring with the task description, generation/candidate info, and the function signature:

    """
Reward function for: Walk forward
Generated by RoboSmith (generation 3, candidate 2)
Score: 45.3 (mean over 5 random episodes)
"""
import numpy as np

def compute_reward(obs, action, next_obs, info):
    forward_velocity = next_obs[13]
    height = next_obs[2]
    velocity_reward = forward_velocity * 2.0
    alive_bonus = 1.0 if height > 0.3 else -10.0
    action_penalty = -0.1 * np.sum(action ** 2)
    total = velocity_reward + alive_bonus + action_penalty
    return total, {"velocity": velocity_reward, "alive": alive_bonus, "action_penalty": action_penalty}

    This file is importable — you can use it directly in your own training scripts.

    report.md¶

    A human-readable Markdown report summarizing the entire run:

    • Task description and parsed specification
    • Environment selected and why
    • Reward design results (generations, best score, reward function summary)
    • Training metrics (algorithm, timesteps, final reward, convergence)
    • Evaluation results (success rate, decision, per-criterion results)
    • Iteration history (if the pipeline looped)
    • Timing breakdown per stage

    eval_report.json¶

    The full EvalReport serialized as JSON, including per-episode results, aggregate metrics, decision, and criteria results. Useful for programmatic analysis.

    policy_rollout.mp4¶

    A video recording of the trained policy performing the task. Recorded by running the policy in the environment with render_mode="rgb_array" and capturing frames.

    The video recording uses a two-tier approach:

    1. Gymnasium's RecordVideo wrapper (preferred) — uses moviepy for clean video output
    2. Frame-by-frame capture with imageio (fallback) — if moviepy isn't installed

    The video typically shows 3–5 episodes concatenated, giving a representative view of the policy's behavior.

    Note

    Video recording requires the [video] extra: pip install -e ".[video]". If video dependencies aren't installed, delivery skips video recording and logs a warning.

    run_state.json¶

    The complete RunState serialized as JSON. Contains every stage's status, duration, metadata, and the full decision history. This is the authoritative record of what happened during the run and is sufficient to understand (and debug) any aspect of the pipeline's behavior.

    task_spec.json¶

    The parsed TaskSpec as JSON. Useful for reproducibility — you can see exactly how the natural language input was interpreted.

    HuggingFace Hub push¶

    If push_to_hub is set in the task spec or config, the delivery stage uploads the checkpoint and report to HuggingFace:

    robosmith run --task "Walk forward" --push-to-hub "username/my-walking-ant"

    Or in config:

    push_to_hub: username/my-walking-ant

    This requires the [hub] extra (pip install -e ".[hub]") and a HuggingFace token set via HUGGING_FACE_HUB_TOKEN or huggingface-cli login.

    How it works¶

    from robosmith.stages.delivery import run_delivery

result = run_delivery(
    state=run_state,
    reward_candidate=best_candidate,
    eval_report=report,
    training_result=training_result,
)

    The delivery stage:

    1. Writes the reward function to reward_function.py
    2. Generates the Markdown report from all stage results
    3. Serializes the eval report and task spec to JSON
    4. Attempts video recording (if env supports rendering and video deps are installed)
    5. Saves the run state
    6. Optionally pushes to HuggingFace Hub

    DeliveryResult¶

    @dataclass
class DeliveryResult:
    artifacts_dir: Path
    files_written: list[str]
    pushed_to_hub: bool = False
    hub_url: str | None = None

    Inspecting run artifacts¶

    After a run, the easiest way to understand what happened:

    # Read the report
cat robosmith_runs/run_*/report.md

# Check evaluation metrics
cat robosmith_runs/run_*/eval_report.json | python -m json.tool

# Look at the reward function
cat robosmith_runs/run_*/reward_function.py

# Watch the video
open robosmith_runs/run_*/policy_rollout.mp4
